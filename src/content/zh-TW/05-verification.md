## 驗證：確保代理不會偏離軌道

代理可以自主行動，但自主不等於不受監督。驗證層的工作是觀察代理的行為，在問題發生之前（或之後）做出反應。

## CostTracker：13 行程式碼的威力

`CostTracker` 可能是 claw-code 中最小但最重要的元件之一。只用 13 行程式碼，它實現了：

- 累積追蹤每次 LLM 呼叫的 token 消耗
- 累積追蹤每次 LLM 呼叫的成本
- 提供即時查詢介面

為什麼如此重要？因為沒有成本追蹤的代理就是一張無限額的信用卡。你永遠不知道下一個回合會花多少錢。

## apply_cost_hook()：最簡單的鉤子模式

`apply_cost_hook()` 只有 8 行程式碼，但它展示了一個強大的設計模式：

- 在每次 LLM 呼叫後觸發
- 檢查累積成本是否超過預算
- 如果超過，中止代理執行

> 鉤子（Hook）模式的精髓：在關鍵節點插入少量程式碼，就能改變整個系統的行為。

## run_parity_audit()：Harness 審計自己

`run_parity_audit()` 是一個自我檢查機制。它的用途是：

- 比較 claw-code 的實作與 Claude Code 的預期行為
- 檢測功能偏差
- 確保重新實作沒有遺漏關鍵邏輯

這是一種元驗證——不是驗證代理的輸出，而是驗證 harness 本身的正確性。

## Rust hooks.rs：完整的鉤子生命週期

claw-code 的 Rust 實作在 `hooks.rs` 中展示了生產級的鉤子系統：

- **PreToolUse**：工具執行前觸發，可以阻止危險操作
- **PostToolUse**：工具執行後觸發，可以審計結果
- **Stop**：代理停止時觸發，用於清理和最終檢查

### 鉤子的設計要點

- 鉤子應該快速執行——它們在關鍵路徑上
- 鉤子應該是可選的——系統沒有鉤子也能運行
- 鉤子應該可組合——多個鉤子可以串聯

## 通用模式

生產代理需要可觀察性和安全閘門。這不是可選功能，而是部署前提：

- **成本追蹤**：知道代理花了多少錢
- **行為鉤子**：在關鍵節點插入檢查邏輯
- **自我審計**：harness 驗證自己的一致性
- **生命週期事件**：在工具使用前後、代理停止時執行額外邏輯

沒有驗證層的代理，不適合在生產環境中使用。
