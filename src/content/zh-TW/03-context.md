## 上下文：代理的工作記憶

LLM 有一個固定大小的上下文視窗。代理如何利用這個有限的空間，直接決定了它的能力上限。上下文管理不是可有可無的功能——它是代理工程的核心挑戰。

## PortContext：工作區掃描

代理啟動時，`PortContext` 負責掃描工作區並收集關鍵資訊：

- 目錄結構
- 專案類型（語言、框架）
- 相關設定檔
- Git 狀態

這些資訊讓代理從第一個回合開始就「了解」你的專案，而不是從零開始摸索。

## build_system_init_message()：組裝系統提示

`build_system_init_message()` 將多個來源的資訊組裝成系統提示：

- 基礎指令（代理的角色和行為規範）
- 工作區上下文（來自 PortContext 的掃描結果）
- 工具定義（當前啟用的工具列表）
- 使用者偏好設定

> 系統提示是代理的「第一印象」。它設定了整個對話的基調和能力邊界。

## 預取模式

預取的核心思想是：在 LLM 發出請求之前，就載入它可能需要的資料。

例如，如果使用者提到了某個檔案，harness 可以在發送提示之前就讀取該檔案的內容，直接放入上下文中。這減少了代理需要的回合數，加快了任務完成速度。

## 訊息壓縮

長時間運行的對話會累積大量訊息，最終超出上下文視窗。claw-code 的壓縮策略是：

- 監控當前上下文的 token 數量
- 當超過閾值時觸發壓縮
- 保留最近 N 條訊息不變
- 將較早的訊息摘要化

壓縮是有損的——你會失去一些細節。但替代方案是完全失去早期上下文，那更糟糕。

## 預算控制

`max_budget_tokens` 是一個硬性上限，防止代理在單次對話中消耗過多 token。

- 追蹤每次 LLM 呼叫的 token 消耗
- 累積計算總用量
- 接近上限時發出警告
- 達到上限時停止代理

沒有預算控制的代理就像沒有油量表的汽車——你不知道什麼時候會耗盡資源。

## 通用模式

長期運行的代理必須管理其上下文視窗。這在每個框架中都會出現：

- 上下文會增長，視窗是固定的
- 壓縮或截斷是必要的妥協
- 預取可以減少回合數，提升效率
- 預算控制防止失控成本
