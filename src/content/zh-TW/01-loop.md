## 代理迴圈：代理的心跳

每個代理的核心都是一個迴圈。這個迴圈異常簡單：

> **觀察 → 決策 → 行動 → 驗證**

然後重複。這就是把被動的語言模型變成主動代理的關鍵。

## 簡單的 while(true)

從概念上來說，代理迴圈就是一個 `while(true)`。Claude Code 的生產版本使用的是無限迴圈——代理持續運行，直到完成任務或被使用者中斷。

但在學習時，無限迴圈不太安全。因此 claw-code 做了一個務實的選擇。

## claw-code 的有限迴圈

在 `runtime.py` 中，`PortRuntime.run_turn_loop()` 實作了一個有限迴圈：

- 預設 `max_turns=3`
- 每個回合呼叫一次 LLM，處理回應，執行工具
- 達到上限後安全退出

這讓你可以安全地實驗，不用擔心代理失控。

## 單次回合的處理

`QueryEnginePort.submit_message()` 負責處理單次回合的完整生命週期：

- 將使用者訊息發送到 LLM
- 接收回應
- 解析工具呼叫
- 返回結果

它的串流變體 `stream_submit_message()` 使用 Python 生成器（generator），讓你可以在回應生成過程中逐步處理 token，而不是等待完整回應。

## route_prompt()：LLM 之前的智慧

在訊息到達 LLM 之前，`route_prompt()` 會先進行一步智慧處理。它的工作原理是：

- 對使用者的輸入進行 token 比對
- 根據比對結果對可用工具進行評分
- 決定哪些工具最可能被需要

這是一種輕量級的預過濾，減少不必要的工具定義傳送給模型，從而節省 token 並提高回應品質。

## 通用模式

觀察-行動迴圈是代理工程中最普遍的模式。無論你使用哪個框架：

- **LangChain** 有 AgentExecutor 迴圈
- **AutoGPT** 有思考-行動迴圈
- **Claude Code** 有 while(true) 迴圈

細節不同，但核心結構完全一致。理解了這個迴圈，你就理解了每個代理框架的骨架。

### 關鍵要點

- 迴圈是代理最基礎的抽象
- 有限迴圈適合學習和測試，無限迴圈適合生產
- 串流處理讓使用者體驗更加流暢
- 路由和預過濾在 LLM 呼叫之前就能提升效率
