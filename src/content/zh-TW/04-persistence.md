## 持久化：跨會話的記憶

對話結束後會發生什麼？沒有持久化的代理每次啟動都從零開始。有了持久化，代理可以記住之前的工作，從上次中斷的地方繼續。

## StoredSession：JSON 儲存

`StoredSession` 負責將會話狀態序列化為 JSON 並儲存到 `.port_sessions/` 目錄：

- 每個會話一個 JSON 檔案
- 包含對話歷史、工具狀態、使用者偏好
- 檔名包含時間戳和會話 ID

儲存格式選擇 JSON 是為了可讀性和可除錯性。當出現問題時，你可以直接打開檔案查看代理的完整狀態。

## from_saved_session()：恢復會話

`from_saved_session()` 實作了「恢復」模式——當使用者使用 `/resume` 指令時：

- 從磁碟讀取儲存的 JSON
- 重建會話狀態
- 恢復對話歷史
- 重新載入工具配置

恢復不是簡單的「讀取檔案」。它需要驗證儲存的狀態是否與當前環境相容，並處理版本差異。

## TranscriptStore：可變訊息列表

`TranscriptStore` 管理對話的訊息列表，特點是：

- 支援訊息的新增和修改
- 內建壓縮支援——可以在不影響對話的前提下壓縮舊訊息
- 是代理「短期記憶」的載體

> TranscriptStore 就像一個智慧筆記本：你可以翻回前幾頁，也可以把早期的筆記濃縮成摘要。

## HistoryLog：僅附加的事件日誌

`HistoryLog` 與 TranscriptStore 不同，它是一個僅附加（append-only）的日誌：

- 記錄代理生命週期中的每一個事件
- 工具呼叫、LLM 回應、錯誤、狀態變更
- 不可修改——一旦寫入就是永久記錄
- 是代理的完整審計軌跡

HistoryLog 的價值在除錯時才真正顯現。當代理行為異常時，你可以回溯每一步操作，找到問題的根源。

### TranscriptStore vs HistoryLog

- **TranscriptStore**：可變的工作記憶，用於構建 LLM 的上下文
- **HistoryLog**：不可變的審計日誌，用於除錯和分析

兩者互補，缺一不可。

## 通用模式

有狀態的代理需要儲存和恢復對話狀態。這個需求產生了兩個通用問題：

- **儲存什麼？** 對話歷史、工具狀態、使用者偏好
- **何時儲存？** 每個回合結束後、使用者中斷時、異常退出時

每個嚴肅的代理框架最終都會實作某種形式的持久化。
