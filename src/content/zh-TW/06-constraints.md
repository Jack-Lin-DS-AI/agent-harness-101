## 約束層

約束定義了代理**不能**做什麼——這是信任的基礎。一個沒有約束的代理就像擁有 root 權限但沒有 sudo 規則的系統：技術上能做任何事，但沒有人敢放心使用。

### ToolPermissionContext：20 行的權限引擎

claw-code 的 `permissions.py` 可能是最具教學意義的文件之一。`ToolPermissionContext` 僅用大約 **20 行 Python 程式碼**就實現了完整的權限引擎。

這個精簡的設計本身就是一個教學重點——安全機制不需要龐大複雜，它需要的是正確和清晰。

### blocks() 方法：比對邏輯

`blocks()` 方法實現了兩種比對策略：

- **名稱精確比對** — 工具名稱是否在拒絕列表中？例如 `Bash` 精確阻擋 `Bash` 工具
- **前綴比對** — 工具名稱是否以拒絕前綴開頭？例如 `mcp__` 可以阻擋所有 MCP 工具

這兩種策略的組合提供了足夠的靈活性。你可以精確地阻擋單一工具，也可以用前綴一次性阻擋整個類別的工具。

### 被拒絕的工具永遠不會到達 LLM

這是約束層最關鍵的設計原則。被標記為拒絕的工具會在工具列表組裝階段就被移除，模型根本不知道這些工具的存在。

這比在工具被呼叫時才拒絕要安全得多。如果模型能在提示中看到工具的描述，它可能會嘗試用各種方式呼叫它。但如果工具完全不在可見範圍內，模型連嘗試的機會都沒有。

### PermissionDenial 追蹤

當一個工具被拒絕時，`PermissionDenial` 會記錄：

- 哪個工具被拒絕了
- 被拒絕的原因是什麼
- 在什麼上下文中發生的

這些記錄在 `TurnResult` 中報告，讓 LLM 知道為什麼它的請求被拒絕。透明度是信任的關鍵——當使用者問「為什麼代理沒有做 X？」時，你可以查看拒絕記錄來找到答案。

### 信任模式

claw-code 支援不同的信任模式：

- **trusted** — 信任模式，完整功能啟用，適合在安全的開發環境中使用
- **restricted** — 限制模式，只允許最基本的安全操作，適合在不確定的環境中使用

信任模式影響 `ToolPermissionContext` 的初始設定——不同模式下，預設被拒絕的工具列表不同。Claude Code 有更豐富的 5 種權限模式，但基本原則相同：在啟動時建立信任邊界。

### 簡潔即力量

值得再次強調：整個權限引擎只有約 20 行。這不是因為功能簡陋，而是因為設計精良。每一行都有明確的目的，沒有多餘的抽象或間接層。

在安全相關的程式碼中，簡潔本身就是一種安全特性。程式碼越短，越容易審查、越不容易藏有錯誤。

> **教學筆記：** 當你設計自己的約束系統時，先從 20 行開始。如果你需要更多，確保你真的需要——而不是在過度工程化。

### 通用模式

安全的代理需要在模型和行動之間設置一個約束層。這個層必須在模型做出決定之後、行動執行之前介入。所有生產級的代理框架都有某種形式的權限系統——差異在於粒度和可配置性。
